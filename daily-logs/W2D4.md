## Commands I Ran

### 1. Generate SSH key pair

```bash
ssh-keygen -t ed25519 -C "your_email_or_label"
ls -la ~/.ssh/
cat ~/.ssh/id_ed25519.pub
```

**Output:**

```
drwx------  2 asia asia 4096 Jul  1 10:37 .
drwx------ 11 asia asia 4096 Jun 30 20:18 ..
-rw-------  1 asia asia  411 Jul  1 10:37 id_ed25519
-rw-r--r--  1 asia asia  101 Jul  1 10:37 id_ed25519.pub
-rw-------  1 asia asia  978 Jun 26 17:32 known_hosts
-rw-r--r--  1 asia asia  142 Jun 26 17:32 known_hosts.old
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIAUMo/2I+rEOFebpCZOGPgItAv4XWQDqWUowfP5g9J+T your_email_or_label
```

**What this does:** Creates a private key (`id_ed25519`, permission 600 — never share) and a public key (`id_ed25519.pub`, permission 644 — safe to share/upload to servers).

### 2. Write SSH config

```bash
nano ~/.ssh/config
chmod 600 ~/.ssh/config
cat ~/.ssh/config
```

**Output:**

```
Host myserver
  HostName 192.168.1.100
  User kali
  IdentityFile ~/.ssh/id_ed25519
  Port 22
```

```
-rw------- 1 asia asia 94 Jul  1 10:44 /home/asia/.ssh/config
```

**What this does:** Sets up an alias `myserver` so I can type `ssh myserver` instead of the full command with IP/user/key path. First attempt had two typos (`mysever` instead of `myserver`, `UdentityFile` instead of `IdentityFile`) — both fixed. Learned that SSH silently ignores unrecognized config keywords instead of throwing an error, so typos in config keys can fail silently.

### 3. Test tunneling syntax (verbose mode)

```bash
ssh -v myserver
```

**Output:**

```
debug1: OpenSSH_10.3p1 Debian-4, OpenSSL 3.6.2 7 Apr 2026
debug1: Reading configuration data /home/asia/.ssh/config
debug1: /home/asia/.ssh/config line 1: Applying options for myserver
debug1: Reading configuration data /etc/ssh/ssh_config
debug1: Reading configuration data /etc/ssh/ssh_config.d/20-systemd-ssh-proxy.conf
debug1: /etc/ssh/ssh_config line 21: Applying options for *
debug1: Connecting to 192.168.1.100 [192.168.1.100] port 22.
debug1: connect to address 192.168.1.100 port 22: No route to host
ssh: connect to host 192.168.1.100 port 22: No route to host
```

**What this does:** Confirms the config file is read correctly and the `myserver` alias resolves to the right HostName/User/IdentityFile/Port. The final "No route to host" error is expected since `192.168.1.100` is a placeholder IP with no real server behind it — proves the config works, just no live target to reach.

### 4. Practice tunneling command syntax (no live target)

```bash
ssh -L 3307:10.0.0.5:3306 admin@1.2.3.4
```

**What this does:** Practiced writing a local port forwarding command: forwards local port 3307 → through jump host 1.2.3.4 → to internal target 10.0.0.5:3306 (e.g. reaching an internal MySQL service through a jump box). No real server existed at this IP, so the command hung/timed out as expected — the goal was syntax only, not a live connection.

## What I Understood

- Private key stays local (permission 600), public key gets shared/deployed to servers
- SSH config lets you alias a full connection (host, user, key, port) under one short name
- SSH does not error on unknown config keywords — it just silently ignores them, so typos can be sneaky bugs
- Local port forwarding syntax: `ssh -L [local_port]:[target_host]:[target_port] user@ssh_server`, and successfully wrote the command correctly for a given scenario without help
- In `ssh -v` debug output, the last line before the final error is what actually matters — earlier lines are just setup/system config noise

## What Still Confused Me
