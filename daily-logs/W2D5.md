## Commands I Ran

### 1. Reversing a hex dump back to binary (Bandit 12)

```bash
xxd -r data.txt > layer1
file layer1
```

**Output:**

```
layer1: gzip compressed data, was "data2.bin", last modified: Wed Jun 24 14:58:58 2026, max compression, from Unix, original size modulo 2^32 578
```

**What this does:** The original file was a hexdump (text representation of binary data). `xxd -r` converts it back into real binary so it can be decompressed.

### 2. Repeated decompress-and-check loop (Bandit 12)

```bash
mv layer1 layer1.gz
gunzip layer1.gz
file layer1
```

Repeated with `.bz2` / `bunzip2` and `.tar` / `tar -xf` as each `file` check revealed a new type, alternating gzip → bzip2 → gzip → tar → tar → bzip2 → tar → gzip, until: **Output:**

```
data8: ASCII text
The password is qQYQiHOBPR8zR61qxYqX45quvihF2uzk
```

**What this does:** Each compression layer has a distinct file signature. `file` reveals the type, then the matching decompression tool is applied. This repeats until plain text is reached.

### 3. Moving an SSH private key between levels (Bandit 13 → 14)

```bash
exit
scp -P 2220 bandit13@bandit.labs.overthewire.org:/tmp/asia_bandit14/sshkey.private ~/sshkey_bandit14
chmod 600 ~/sshkey_bandit14
ssh -i ~/sshkey_bandit14 bandit14@bandit.labs.overthewire.org -p 2220
```

**What this does:** Bandit blocks localhost-to-localhost logins between levels, so the private key had to be copied out to the local WSL2 machine first (via `scp`), given correct permissions (`chmod 600`), and then used to SSH back in from outside the session — not from inside the bandit13 shell.

### 4. Submitting a password to a plain TCP port (Bandit 14)

```bash
cat /etc/bandit_pass/bandit14
nc localhost 30000
```

**Output:**

```
aaWecNkG4FhxJQxz07uiwzVP6bJiYS65
Correct!
pbLYuZtTg4MgaqfJx8jbA9gKKGqM68A7
```

**What this does:** Every level's own password is stored at `/etc/bandit_pass/bandit<N>`, readable only by that user. Sending it to the listening service on port 30000 with `nc` returns the next level's password.

### 5. Submitting a password over SSL/TLS (Bandit 15)

```bash
openssl s_client -connect localhost:30001
```

**Output:**

```
... certificate handshake output ...
pbLYuZtTg4MgaqfJx8jbA9gKKGqM68A7
Correct!
kS0Hf0u5HiXFwKMKFqXvPdOTNGGa0X8V
```

**What this does:** Port 30001 requires an SSL/TLS-encrypted connection, which plain `nc` doesn't support. `openssl s_client -connect` performs the SSL handshake first, then allows the password to be typed in the same way as `nc`.

---

## What I Understood

- The decompress loop is always the same pattern: `file` → match tool to type → decompress → repeat until "ASCII text".
- Private keys retrieved inside a Bandit level must be copied out to the local machine before being used to log into the next level — logging in from inside the same session (localhost) is blocked.
- When a level's goal mentions "SSL" or "encryption," swap `nc` for `openssl s_client -connect host:port`; the interaction pattern (connect → type password → get next password) stays the same.
